# Unity Migration Plan

This branch (`unity-port`) is where a Unity/C# rewrite of Wasteland Tactics
will live, kept separate from the working pygame version on `main`. Nothing
has been built yet — this document is the agreed-upon plan before any code
gets written.

## Why consider Unity at all

The pygame version works and is fully playable, so this isn't about fixing
something broken. The case for Unity is distribution: Unity's Build Settings
can export to Windows/Mac/Linux **and WebGL**. A WebGL build means a "demo"
can be a link someone clicks and plays in a browser, with zero install —
something the current "clone the repo, install uv, run from source" setup
can't offer. Unity also has native isometric Tilemap support and a real
sprite/animation pipeline, which would replace a fair amount of hand-rolled
projection math and procedural-sprite code.

The honest cost: it's a full rewrite in C#, not a port, and it's a different
day-to-day development loop than the terminal-driven, headless-Python-script
iteration this project has used so far.

## Project structure

```
Assets/
  Scripts/
    Core/          # Unit, BattleGrid, turn queue - plain C#, no MonoBehaviour
    Overworld/      # OverworldController, movement, encounters
    Battle/         # BattleController, AI, action resolution
    Rendering/      # camera rotation, sprite-facing selection
  Sprites/          # PixelLab exports go here directly (Unity imports natively)
  Animations/       # Animator Controllers per character
  Tilemaps/         # Grid + Isometric Tilemap assets
  Scenes/
    Overworld.unity
    Battle.unity
ProjectSettings/    # generated by the Unity Editor, not hand-authored
```

## System-by-system mapping

| Current (pygame)                                              | Unity equivalent                                                                             | Notes                                                                                                                                                                                       |
| --------------------------------------------------------------- | ----------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `game/iso.py` projection math                                  | Unity's built-in **Isometric Tilemap** (Grid component, Isometric layout)                       | The diamond math, depth-sorting, and bounds-fitting we hand-rolled become largely unnecessary - Unity does this natively. `rotate_coords`/`screen_relative_delta` logic is still needed for camera-relative movement, but shrinks to a small utility class since Unity's `Grid.CellToWorld` replaces `grid_to_screen`. |
| `game/entities.py` (`Unit` dataclass)                           | Plain C# class (`Unit`), stats possibly authored as `ScriptableObject` templates                 | Direct conceptual port - same fields, same `roll_damage`/`take_damage` logic.                                                                                                              |
| `game/overworld.py`                                             | `OverworldController : MonoBehaviour` + Tilemap painted with the same tile legend                | The ASCII map in `maps.py` either gets hand-painted into a Tilemap in the Editor, or a one-time importer script reads the ASCII and paints it - keeps the data-driven map idea.            |
| `game/battle.py` (grid, turn queue, BFS movement range, AI)     | `BattleController : MonoBehaviour` orchestrating plain C# logic classes                          | Ports almost 1:1 conceptually - BFS reachable-tiles, turn queue, phase state machine are algorithms, not pygame-specific.                                                                  |
| `game/sprites.py` (procedural + PixelLab loader)                | Native Unity `Sprite` / `Animator` - procedural fallback likely dropped entirely                  | Unity's sprite pipeline replaces both halves of this file; directional rotations become an 8-direction `Animator` blend tree or a simple direction -> clip switch.                          |
| `main.py` state machine                                         | Unity scene management or a `GameStateManager` singleton                                          | Overworld/Battle become separate scenes (or one scene with enable/disable) instead of a hand-written state enum.                                                                          |

## Verification / testing strategy

Unity has a built-in Test Framework (NUnit-based) that runs in **batch mode
from the command line**, no Editor GUI required - so the core logic (turn
queue, BFS, damage rolls, rotation math) can get the same kind of headless,
scripted verification this project has relied on with pygame + screenshots.

There's also a real option for driving the Editor itself via MCP:

- **Unity's official MCP** - part of the AI Assistant package, requires a
  Unity subscription (doesn't consume separate AI credits)
- **Community options built for Claude Code** - free and open source:
  - https://github.com/codergamester/mcp-unity
  - https://github.com/IvanMurzak/Unity-MCP
  - https://github.com/CoplayDev/unity-mcp

These give an agent scene/asset management, script editing, running tests,
reading console output for compile errors, starting/stopping Play Mode, and
capturing screenshots for visual verification - close to the headless
iterate-and-verify loop used on the pygame version.

**Caveat:** these bridge to a Unity Editor that has to actually be running -
the MCP server is a plugin living inside the Unity project, not a cloud
endpoint. This means it only becomes usable once development happens with
Claude Code running locally (on a machine with Unity Hub/Editor installed),
not from this sandboxed remote environment, which has no Unity install and
can't reach an Editor running on a local machine.

## CI / release pipeline

`game-ci/unity-builder` is the standard GitHub Action for headless Unity
builds - it runs the Editor in a Docker container in CI. Plan:

1. Build target: Linux and WebGL
2. Trigger: on tag push (same release workflow already used on `main`)
3. Publish the resulting build artifacts straight to a GitHub Release

One-time setup cost: Unity requires a license file even for CI. The
`game-ci` project documents the flow - request an activation file from a
headless Unity instance, submit it to Unity to get a license file, store
that as an encrypted repo secret.

This also sidesteps a real limitation hit repeatedly on `main`: this
session's git integration can push branches but not tags, and has no
release-creation tool. A CI-driven release uses GitHub Actions' own
credentials, not this chat session's integration, so tag/release automation
here would actually work end-to-end.

## Suggested phase order

1. Port core data/logic only (`Unit`, `BattleGrid`, turn queue, BFS) with
   NUnit tests - no rendering yet, verify it's behaviorally identical to the
   Python version
2. Overworld scene: Tilemap + movement + camera rotation
3. Battle scene: grid rendering, move/attack highlighting, click-to-act
4. Wire in the existing PixelLab sprite assets
   (`assets/characters/wasteland_survivor_wearing_a_faded/`)
5. CI build pipeline + first automated WebGL demo release
